# Research: Recommendation Provider Retry Mechanism

## Decision 1: Where to implement retry logic

**Decision**: Standalone utility function `post_with_retry` in `src/recommendation/retry.py`, called from each adapter.

**Rationale**: All four adapters (Service1, Service2, Service3, Dynamic) share the same pattern — build a request body, call `client.post(url, json=body)`, check the status. A standalone async function that accepts `(client, url, json_body, provider_id)` can be reused by all without modifying the `ProviderAdapter` ABC or adding httpx coupling to the base class. Adding it to the ABC would require the base class to know about `self._client`, which is an implementation detail of concrete adapters.

**Alternatives considered**:
- *Retry in aggregator `_call`*: The aggregator doesn't know each adapter's endpoint URL, so it cannot produce the required log message (endpoint + provider name + HTTP status). Rejected.
- *Retry as base class method*: Would couple `ProviderAdapter` to `httpx.AsyncClient`. Rejected.
- *Retry in each adapter individually*: 4× duplication of identical logic. Rejected.

---

## Decision 2: Which HTTP errors trigger retry

**Decision**: Retry on HTTP 5xx and connection/network exceptions only. Return immediately on HTTP 4xx.

**Rationale**: 4xx errors (Bad Request, Unauthorized, Not Found) indicate a problem with the request data or caller credentials — retrying with identical data will not produce a different result and could trigger rate limiting. 5xx errors (Internal Server Error, Service Unavailable, Gateway Timeout) are transient server-side failures where a retry is likely to succeed.

**Alternatives considered**:
- *Retry all non-2xx*: Would retry 4xx needlessly and risk rate-limit penalties. Rejected.
- *Retry only network exceptions*: Would miss HTTP 503/502 which are common transient failures. Rejected.

---

## Decision 3: Request body construction and retry consistency

**Decision**: Build the full request body dict **once before the retry loop**. Pass the pre-built dict to `post_with_retry`. This includes session tokens (UUIDs) generated by Service2/Service3.

**Rationale**: The spec requires "retries with the same data". Generating a new UUID on each attempt would send different data on each retry, violating this requirement. Constructing the body once and reusing it across all attempts satisfies the spec and is simpler.

**Alternatives considered**:
- *Generate new UUID per attempt*: Different data per attempt. Violates FR-002. Rejected.

---

## Decision 4: Timeout interaction

**Decision**: The `asyncio.wait_for` timeout in the aggregator (800ms default) governs the **entire** call including all retries. No per-attempt sub-timeout is introduced.

**Rationale**: Adding a per-attempt timeout would require configuring a separate value and changes to `post_with_retry`'s interface. The existing 800ms budget already bounds worst-case latency. If retries are needed they will naturally be cut off by the outer timeout. The spec Assumptions section explicitly documents this decision.

---

## Decision 5: Log structure

**Decision**: Use structured log at `ERROR` level via the existing `structlog`-based `get_logger`. Fields: `provider`, `endpoint`, `http_status` (int or None), `error` (exception string or None), `attempts` (int).

**Rationale**: Consistent with all existing error log calls in the codebase (e.g. `logger.error("service1_error", error=str(exc))`). Adding `endpoint` and `attempts` fields gives the operator everything needed to diagnose the failure without grepping multiple sources.
